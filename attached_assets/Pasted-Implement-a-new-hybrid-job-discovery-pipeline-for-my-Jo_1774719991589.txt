Implement a new hybrid job discovery pipeline for my Job Search Scout app.

Goal:
Keep JobSpy as the primary scraping backbone for job boards, but add a second discovery/enrichment layer using the Gemini API with Grounding with Google Search so it can mimic Google AI Search behavior: real-time web-grounded responses with citations, surfaced sources, and search metadata. Gemini should help find additional job listings and direct company career pages that JobSpy may miss, then enrich those jobs with better metadata and descriptions. The backend must remain the final source of truth for matching jobs to user settings. Do NOT let Gemini or Claude directly decide what jobs appear on the jobs page without deterministic backend filtering.

Core product rule:

* User settings in the frontend already define what JobSpy and Gemini should search for.
* JobSpy + Gemini are discovery inputs.
* Backend deterministic logic is the gatekeeper.
* Claude is downstream only for tailoring/writing.

High-level architecture:

1. User settings define exact job criteria.
2. JobSpy scrapes jobs from supported sources.
3. Gemini with Google Search grounding discovers additional jobs and direct company career pages.
4. Normalize both JobSpy jobs and Gemini jobs into one common schema.
5. Deduplicate aggressively.
6. Apply deterministic hard filters based on user settings.
7. Apply a scoring/ranking layer.
8. Only then pass selected jobs and their descriptions to Claude for resume tailoring / downstream writing.

Important principle:

* Backend logic decides what jobs qualify.
* Gemini is only for discovery + enrichment.
* Claude is only for resume tailoring / candidate-facing writing.
* Do NOT build a pipeline where raw Gemini results and raw JobSpy results are dumped into Claude and Claude decides what fits.

PART 2: GEMINI SEARCH INTEGRATION
Add a Gemini-based discovery module that uses Gemini API + Grounding with Google Search to find additional job listings and direct company career pages based on the saved user settings.

Gemini requirements:

* Read Gemini API key from Replit Secrets / environment variables.
* Use the current Google GenAI SDK / currently supported Gemini API client for this project stack. Do not assume an outdated SDK.
* Prefer model selection via env var, with recommended defaults:

  * GEMINI_MODEL=gemini-3.1-pro-preview for best quality / complex reasoning
  * fallback option GEMINI_MODEL=gemini-3-flash-preview for lower latency / lower cost
* Use the current Google Search grounding tool name for current Gemini models:

  * `google_search`
* Do NOT use `google_search_retrieval` unless the installed SDK/version truly requires it for an older model path.

Use Gemini search only for:

* finding jobs JobSpy may miss
* finding direct company careers pages
* validating job URLs/pages are still live
* extracting or enriching job descriptions/metadata
* generating richer company/job context for the jobs report

Do NOT use Gemini as the final decision-maker for fit.

Search strategy:
Generate selective web queries based on saved user settings. Examples:

* "remote enterprise account executive site:greenhouse.io OR site:lever.co OR site:ashbyhq.com"
* "account executive careers remote SaaS"
* "site:company.com/careers enterprise sales remote"
* "site:jobs.ashbyhq.com senior account executive remote"
* "site:boards.greenhouse.io enterprise account executive"

Use settings to dynamically generate a focused set of search queries rather than generic wide-net web search.

PART 5: DEDUPLICATION
Build dedupe logic before ranking. Deduplicate across JobSpy and Gemini results using:

* normalized URL
* company + normalized title
* company + location + title
* fuzzy similarity for reposted duplicates
* prefer the better-quality record when duplicates exist:

  * keep the version with fuller description
  * keep the direct company URL when possible
  * keep the fresher posted date when possible
  * merge useful metadata from both records

Return a single deduped canonical job record.

PART 7: SCORING / RANKING LAYER
After hard filtering, add a ranking/scoring layer.

Build two scores:

1. deterministic_score
2. llm_score

Deterministic score should consider:

* title match strength
* keyword match strength
* remote/location match
* salary match
* company whitelist boost
* freshness
* source quality
* direct company site boost
* description completeness

LLM score should be optional and used only after the hard filters pass.
The LLM score should evaluate nuanced fit like:

* how aligned the role is to the user’s broader goals
* whether the role looks strategically strong
* whether the description suggests likely fit or stretch
* whether the posting looks low quality / evergreen / suspicious
* short fit explanation

If you implement LLM scoring, keep it structured and bounded. Do not let it overrule hard disqualifiers.

Final ranking should combine deterministic score + optional llm_score.
