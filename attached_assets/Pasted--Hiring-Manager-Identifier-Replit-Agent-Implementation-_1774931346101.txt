# Hiring Manager Identifier — Replit Agent Implementation Prompt

## OVERVIEW

Add a "Hiring Manager Identifier" capability to the existing Job Scout Agent. This feature automatically identifies the most likely hiring manager for any saved job, and displays that information on both the **Saved Jobs** page and the **My Pipeline** page. It uses Google Custom Search API to find LinkedIn profiles, then Claude Opus 4.6 to analyze results and pick the right person.

### New API Endpoint: `/api/hiring-manager/identify`

**Method**: POST

**Request body**:
```json
{
  "job_id": 123,
  "company_name": "Akamai",
  "role_title": "Strategic Account Executive - Southeast",
  "job_description": "...(full JD text)...",
  "location": "Southeast US"
}
```

**What this endpoint does (3-step process):**

#### Step 1: Claude Analyzes the Job Description

Call Claude Opus 4.6 (model: `claude-opus-4-6`) WITHOUT web search to extract structured info from the JD.

```typescript
const analysisResponse = await fetch("https://api.anthropic.com/v1/messages", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "x-api-key": process.env.ANTHROPIC_API_KEY,
    "anthropic-version": "2023-06-01"
  },
  body: JSON.stringify({
    model: "claude-opus-4-6",
    max_tokens: 1024,
    system: `You analyze job descriptions to determine who the hiring manager likely is. Given a job description, extract the following and return ONLY valid JSON with no other text:

{
  "department": "the department this role sits in (e.g., Enterprise Sales, Security Sales, Channel Sales, Cloud Sales)",
  "role_seniority": "the seniority of the OPEN role (e.g., AE, Senior AE, Director)",
  "manager_likely_title": "the title of the person this role most likely reports to (e.g., if the role is AE, the manager is likely VP Sales, Director of Sales, or Regional Sales Director)",
  "manager_seniority_keywords": ["array of 3-5 title keywords to search for, e.g., 'VP Sales', 'Director Enterprise Sales', 'Head of Sales', 'Regional Sales Director', 'AVP Sales'"],
  "region": "the geographic region or territory mentioned",
  "vertical": "any industry vertical mentioned (e.g., healthcare, financial services, technology)",
  "named_people": ["any names explicitly mentioned in the JD, usually recruiters"]
}

Rules:
- If the role is an AE or Account Executive, the hiring manager is typically a Director, VP, or Regional Director of Sales
- If the role is a Director, the hiring manager is typically a VP or SVP
- If the role is a Sales Manager, the hiring manager is typically a Director or VP
- Be specific about the sales segment: "Enterprise Sales" not just "Sales"
- Include the region in your manager title keywords when a region is specified
- Return ONLY the JSON. No markdown, no explanation, no backticks.`,
    messages: [
      {
        role: "user",
        content: `Company: ${companyName}\nRole: ${roleTitle}\n\nJob Description:\n${jobDescription}`
      }
    ]
  })
});
```

Parse the JSON response. If parsing fails, try stripping markdown backticks and re-parsing.

#### Step 2: Google Custom Search

Using the `manager_seniority_keywords` from Step 1, run 2-3 Google Custom Search queries. Use the Google Custom Search JSON API:

```typescript
// Build 2-3 search queries from the analysis
const searchQueries = [
  `"${analysis.manager_likely_title}" site:linkedin.com/in ${companyName}`,
  `"${analysis.manager_seniority_keywords[0]}" OR "${analysis.manager_seniority_keywords[1]}" site:linkedin.com/in ${companyName}`,
  `${companyName} ${analysis.region || ''} sales leadership site:linkedin.com/in`
];

// Run each query against Google Custom Search API
for (const query of searchQueries) {
  const googleResponse = await fetch(
    `https://www.googleapis.com/customsearch/v1?key=${process.env.GOOGLE_CSE_API_KEY}&cx=${process.env.GOOGLE_CSE_ID}&q=${encodeURIComponent(query)}&num=5`
  );
  const googleData = await googleResponse.json();
  // Collect all results into an array
  allResults.push(...(googleData.items || []));
}
```

**IMPORTANT**: You need TWO environment variables for Google Custom Search:
- `GOOGLE_CSE_API_KEY` — the API key (user already has this)
- `GOOGLE_CSE_ID` — the Custom Search Engine ID (user already has this)

Collect all Google results (titles, snippets, URLs) into a single text block.

#### Step 3: Claude Identifies the Hiring Manager

Send the Google results + the JD analysis to Claude Opus 4.6 to pick the most likely hiring manager:

```typescript
const identifyResponse = await fetch("https://api.anthropic.com/v1/messages", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "x-api-key": process.env.ANTHROPIC_API_KEY,
    "anthropic-version": "2023-06-01"
  },
  body: JSON.stringify({
    model: "claude-opus-4-6",
    max_tokens: 1024,
    system: `You are a hiring manager identification engine. Given Google search results (mostly LinkedIn profiles) and context about a job opening, identify the single most likely hiring manager.

Rules:
- The hiring manager is the person this role REPORTS TO, not the recruiter
- Ignore recruiters, talent acquisition, HR people — they post the job but don't manage the role
- Ignore people with the SAME title as the open role — they'd be peers, not managers
- Look for someone one level ABOVE the open role in the same department and region
- If the role is "Account Executive - Southeast", look for a "Director of Sales" or "VP Sales" who covers Southeast or is based in a Southeast city
- If multiple candidates match, pick the one whose title and region most closely align
- If no strong match exists, say so honestly — don't guess

Return ONLY valid JSON with no other text:

{
  "full_name": "First Last",
  "title": "Their current title at the company",
  "linkedin_url": "https://linkedin.com/in/their-profile",
  "confidence": "high/medium/low",
  "reasoning": "One sentence explaining why you picked this person",
  "alternative": {
    "full_name": "Second choice name or null",
    "title": "Second choice title or null",
    "linkedin_url": "URL or null"
  }
}

Confidence levels:
- HIGH: Title is clearly one level above the open role, same department, same region, currently at the company
- MEDIUM: Title matches but region is unclear, or the person might be lateral, or the profile snippet is ambiguous
- LOW: Best guess from limited data — might be wrong

If you truly cannot identify anyone from the results, return:
{
  "full_name": null,
  "title": null,
  "linkedin_url": null,
  "confidence": "none",
  "reasoning": "Explanation of why no match was found",
  "alternative": null
}`,
    messages: [
      {
        role: "user",
        content: `OPEN ROLE CONTEXT:
Company: ${companyName}
Open Role: ${roleTitle}
Department: ${analysis.department}
Role Seniority: ${analysis.role_seniority}
Expected Manager Title: ${analysis.manager_likely_title}
Region: ${analysis.region || 'Not specified'}
Vertical: ${analysis.vertical || 'Not specified'}
Named People in JD: ${analysis.named_people?.join(', ') || 'None'}

GOOGLE SEARCH RESULTS:
${allResults.map((r, i) => `Result ${i + 1}:\nTitle: ${r.title}\nSnippet: ${r.snippet}\nURL: ${r.link}`).join('\n\n')}`
      }
    ]
  })
});
```

### Google CSE Configuration Note

Your Google Custom Search Engine should be configured to search the entire web (not restricted to specific sites). The `site:linkedin.com/in` restriction is applied in the search query itself, not in the CSE config. This way the same CSE can also search for company info, news, etc.

If your CSE is currently restricted to specific sites, go to https://programmablesearchengine.google.com and change "Search the entire web" to ON.

## PART 3: HOW IT WORKS END-TO-END

Here's what happens when you click "Identify Hiring Manager" on a saved job:

```
1. You click the button
   ↓
2. Frontend sends job_id, company, role, JD to /api/hiring-manager/identify
   ↓
3. Backend sends JD to Claude Opus 4.6 → Claude returns structured analysis:
   "This AE role reports to a VP/Director of Enterprise Sales in the Southeast"
   ↓
4. Backend builds 2-3 Google search queries using Claude's analysis:
   → '"VP Enterprise Sales" site:linkedin.com/in Akamai'
   → '"Director Sales" Southeast site:linkedin.com/in Akamai'
   ↓
5. Google returns LinkedIn profile snippets (names, titles, URLs)
   ↓
6. Backend sends Google results + JD context to Claude Opus 4.6
   → Claude picks the most likely hiring manager and explains why
   ↓
7. Backend saves to database, returns to frontend
   ↓
8. Card updates to show: "Jane Smith — VP Enterprise Sales, Southeast"
   with a green "High Confidence" badge and a LinkedIn link
```

## PART 4: ACCURACY TIPS

This system won't be 100% accurate. Here's what to expect:

**High accuracy scenarios (80-90%)**:
- Large public companies (Akamai, Dell, Cisco) with lots of LinkedIn profiles
- Sales roles where the management hierarchy is standard (AE → Director → VP)
- US-based roles with clear region/territory in the title

**Lower accuracy scenarios (50-60%)**:
- Small startups where the CEO might be the hiring manager
- Roles that could report to multiple departments (e.g., Sales Engineer could report to Sales VP or Engineering VP)
- Companies where everyone has inflated titles
- International roles

**Why manual override matters**: The "Edit" button isn't a fallback — it's a core feature. Even when the AI gets it wrong, it usually narrows it down to 2-3 candidates. The user picks the right one and saves it. Over time, the confirmed data in your database becomes a valuable asset.
